Java 线程间的协作

wait/notify/notifyAll

一定要和synchronized配合使用，实例代码：leetcode1115，交替打印FooBar。

题意大概是有两个线程分别调用同一个对象的foo()和bar()，需要按顺序输出。

Object::wait/notify/notifyAll方法构成了内部条件队列的API。正如每个java对象都可以作为一个锁，每个对象同时可以作为一个条件队列。条件队列使得一组线程能够通过某种方式来等待特定的条件变为真，这个队列里的元素是一个个正在等待相关条件的线程。

对象的内置锁与其内部条件队列是相互关联的，要调用对象X中条件队列的任何一个方法，必须持有对象X上的锁。这是因为“等待由状态构成的条件”与“维护状态一致性”这两种机制必须紧密地绑定在一起：只有能对状态进行检查时，才能够在某个条件上等待，并且只有能修改状态时，才能从条件等待中释放另一个线程。从代码角度来看，如果上述代码没有synchronized关键字，那么count的值并不会得到保证——可能同时调用foo方法两次——这样，整个程序代码的正确性（交替打印）也得不到保证。

内部条件队列有以下缺点，可以通过正确的编程方式来避免

  • 过早唤醒：指的是wait()方法返回后，并不能保证条件谓词（上述的代码中while里的条件）一定为真。因为每次notifyAll时，只是将条件队列里面等待的线程唤醒，可能已经有其余线程已经获取了锁，重新更改了条件谓词，所以需要用while循环检查。在上面的例子中，假设线程A调用foo方法，线程B调用bar方法，并且B线程已经执行到this.wait()方法进入阻塞状态了，当foo方法满足条件执行完this.notifyAll()之后，会将B线程从阻塞状态唤醒，然后AB线程同时竞争锁，此时仍然可能时A线程获取锁——我们期望此时是B线程获取锁，但是这个行为并不能得到保证，单独看bar方法，如果A线程调用的foo方法不会重新调用this.wait进入阻塞状态，那么count的值仍然可能会被改变，所以在bar方法中，一定需要有一个循环检查。同理foo方法也类似。notify和notifyAll的选择也是类似，使用notifyAll可以更加容易的保证程序的正确性。

  • 信号丢失：如果调用wait方法之前没有检查条件谓词，就会出现信号丢失问题；

Condition

和内置的条件类似，每个condition绑定一个lock，但是condition/lock体系比内置的synchronized/wait/notify/notifyAll体系是更加灵活、功能更多的体系：

  • 支持可中断；

  • 支持公平锁；

  • 每个锁可以绑定多个condition；

  • 基于时间的等待

CyclicBarrier

可循环屏障：让一组线程达到同步点是被阻塞，知道最后一个线程达到屏障时，才会允许所有线程同时通过。

CyclicBarrier的实现没有实现AQS框架，而是利用Lock锁+count变量+condition条件队列来控制等待的线程，在代码实现角度来看，当count变量减少至0时（在运行正常的情况下），会重新开始一个循环。

CountDownLatch和CyclicBarrier的区别：

  1. 作用不一样，CountDownLatch主要是一个线程等待另外一组线程全部执行一组操作才能够进行下一步操作；而CyclicBarrier则是所有线程全部执行完一组操作才能够进行下一步操作。

  2. CountDownLatch是不可循环使用的，而CyclicBarrier是可以循环使用的。

阻塞队列

和普通的队列相比，额外提供了阻塞式获取/插入操作，各种在队列不可用的情况下的操作如下：

阻塞队列支持生产者-消费者模式。生产者-消费者模式可以将生产逻辑和消费逻辑解耦开来，两者仅仅依赖共享工作队列进行通讯。

阻塞队列的类型有：

  • ArrayBlockingQueue：数组构成、有界、FIFO、默认不公平；

  • LinkedBlockingQueue：链表构成、有界、FIFO、默认不公平；

  • PriorityBlockingQueue：支持优先级排列的无界队列，和Arrays.sort类似，可以通过实现comareTo()方法来实现优先级；

  • DelayQueue：使用优先级队列实现的无界阻塞队列，元素必须实现Delayed接口；

  • SynchronousQueue：一个不存储元素的阻塞队列；

  • LinkedTransferQueue：链表构成、无界；优先将元素直接传递给阻塞获取的消费者，否则放入队列尾部

  • LinkedBlockingDeque：链表构成，双向队列。可以使用在工作密取的模式中。

    工作密取：通常是每个消费者都有自己的一个消费双端队列，如果自身队列里面的元素没有了，可以从尾部拿到其余消费者队列里的任务执行。

阻塞队列（以ArrayBlockingQueue为例）实现，主要是通过维护一个Lock锁+两个条件队列来实现的。
